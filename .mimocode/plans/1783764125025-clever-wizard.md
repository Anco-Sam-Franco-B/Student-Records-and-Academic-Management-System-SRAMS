# SRAMS Project Analysis & Study

## Project Overview

**SRAMS** (Student Records and Academic Management System) is a full-stack web application designed for educational institutions to manage student records, academic operations, and administrative tasks.

---

## 1. Technology Stack

### Backend (Node.js/Express)
- **Runtime**: Node.js with ES Modules
- **Framework**: Express.js v5.2.1
- **Database**: PostgreSQL (via `pg` library)
- **Authentication**: JWT (jsonwebtoken) with cookie-based sessions
- **Email**: Nodemailer with SMTP
- **Security**: Helmet, CORS, bcrypt for password hashing
- **File Upload**: Multer
- **Scheduling**: node-cron
- **Logging**: Winston with daily rotate file
- **Templates**: Handlebars for email templates

### Frontend (React)
- **Framework**: React 19.2.7 with Vite
- **Routing**: React Router DOM v7
- **State Management**: Zustand (available but not heavily used - mostly local state)
- **Styling**: Tailwind CSS + DaisyUI
- **HTTP Client**: Axios with interceptors
- **PDF Generation**: @react-pdf/renderer, jspdf, html2canvas
- **Animations**: Framer Motion, AOS
- **PWA Support**: vite-plugin-pwa

## 3. Database Schema Analysis

### Core Tables (16 tables)

| Table | Purpose | Key Relationships |
|-------|---------|-------------------|
| `roles` | User roles (Administrator, Teacher, Student, Parent) | Referenced by users |
| `trade` | Academic departments/programs (code + name) | Referenced by classes, students, subjects, teachers |
| `users` | System users with authentication | Links to roles, teachers |
| `students` | Student records with admission numbers | Links to classes, trade |
| `teachers` | Teacher profiles | Links to users, trade |
| `classes` | Class groups within trades | Links to trade |
| `subjects` | Curriculum subjects | Links to trade, classes |
| `teacher_subjects` | Teacher-subject-class assignments (junction table) | Many-to-many relationship |
| `academic_years` | Academic calendar years | Referenced by terms |
| `terms` | Academic terms within years | Links to academic_years |
| `assessments` | Exams and continuous assessments | Links to teachers, subjects, terms |
| `marks` | Student scores on assessments | Links to students, assessments, teachers |
| `attendance` | Daily attendance records | Links to students, teachers, classes |
| `grading_system` | Grade boundaries (A, B, C, etc.) | Standalone |
| `report_cards` | End-of-term student reports | Links to students, terms |
| `student_promotions` | Class progression records | Links to students, classes |
| `notifications` | System notifications | Links to users |
| `database_backups` | Backup metadata | Standalone |

### Key Relationships
- **Trade-centric design**: Most entities (classes, subjects, students, teachers) are organized by "trade" (academic department)
- **Hierarchical**: Academic Year вҶ’ Terms вҶ’ Assessments вҶ’ Marks
- **Junction table**: `teacher_subjects` handles many-to-many teacher assignments across trades and classes

## 6. Authentication & Security

### Authentication Flow
1. User registers вҶ’ Password hashed with bcrypt вҶ’ Stored in `users` table
2. User logs in вҶ’ JWT created вҶ’ Set as httpOnly cookie (`accessToken`)
3. Protected routes check cookie вҶ’ `authenticate` middleware verifies JWT
4. Logout clears cookie

### Security Features
- **CORS**: Strict origin whitelist (localhost:5173)
- **Helmet**: HTTP security headers
- **JWT**: Token-based authentication with expiration
- **Password hashing**: bcrypt with 10 salt rounds
- **Cookie security**: httpOnly, sameSite=lax

### Security Concerns Identified
1. **SQL Injection risk**: `data.controller.js` uses string interpolation for table names (mitigated by `allowedTables` whitelist)
2. **No rate limiting**: Login endpoint vulnerable to brute force
3. **No input validation**: Most endpoints lack request body validation
4. **Missing authorization**: No role-based access control on endpoints (any authenticated user can access all data)
5. **secure: false** on cookie in development (acceptable for dev)

## 7. Code Quality Observations

### Strengths
- Clean separation of concerns (routes, controllers, middleware)
- Consistent error handling patterns
- Proper database transaction usage (admission numbers, role changes)
- Good startup sequence with progress feedback
- Email template system with Handlebars

### Issues to Address

#### Backend
1. **No input validation library** (e.g., Joi, express-validator) - raw req.body usage
2. **Inconsistent error responses** - some use `message`, some use `error`
3. **Missing DELETE routes** for most resources
4. **Student controller** (`student.controller.js`) exists but is not used (data.controller handles everything)
5. **Cron jobs** are stub implementations (just console.log)
6. **No pagination** on list endpoints

#### Frontend
1. **Console.log statements** left in production code (Students.jsx:77)
2. **Unused imports** in some files
3. **Inconsistent error handling** - some use alert(), some use toast
4. **Missing edit functionality** on most list pages
5. **No loading states** on form submissions in some pages
6. **Subjects.jsx** has useEffect dependency issues (line 59)

## 9. Recommendations for Improvement

### High Priority
1. **Add input validation** - Use Joi or express-validator on all endpoints
2. **Implement role-based access control** - Differentiate admin vs teacher vs student permissions
3. **Add missing CRUD operations** - Edit and delete for all entities
4. **Complete attendance marking flow** - Teacher marks attendance вҶ’ stored in DB
5. **Implement marks entry** - Teacher enters marks вҶ’ stored вҶ’ calculated for reports

### Medium Priority
6. **Add pagination** - For all list endpoints to handle large datasets
7. **Implement search/filter** - Connect frontend search inputs to backend queries
8. **Add rate limiting** - Protect login and sensitive endpoints
9. **Standardize API responses** - Consistent format across all endpoints
10. **Implement report card generation** - Calculate averages, positions, generate PDF

### Low Priority
11. **Add WebSocket notifications** - Real-time updates instead of polling
12. **Implement student/teacher portals** - Route StudentLayout and TeacherLayout
13. **Add data export** - CSV/Excel export for reports
14. **Add audit logging** - Track who made changes
15. **Add unit tests** - Critical paths like auth, marks calculation

## Summary

SRAMS is a well-structured school management system with solid foundations in authentication, database design, and basic CRUD operations. The main gaps are in the **academic workflow features** (attendance marking, marks entry, report generation) which have database schemas but lack implementation. The codebase is clean and follows good practices, but needs input validation, proper error handling standardization, and completion of the core academic features to be production-ready.
